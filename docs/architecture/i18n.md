# Internationalization (i18n)

## User Behavior

- Visiting `/` shows content in German (default locale)
- Visiting `/en/...`, `/ru/...`, `/ar/...` shows content in the matching locale
- First-time visitors are auto-redirected based on their browser's `Accept-Language` header
- A `locale` cookie remembers the user's choice across visits
- A language switcher allows manual locale selection
- Arabic renders right-to-left (`<html dir="rtl">`); German, English and Russian render LTR

## Supported Locales

| Code | Name    | Native  | Direction |
| ---- | ------- | ------- | --------- |
| `de` | German  | Deutsch | LTR       |
| `en` | English | English | LTR       |
| `ru` | Russian | Русский | LTR       |
| `ar` | Arabic  | العربية | **RTL**   |

`de` is the default — `/` and `/{path}` resolve to German with no prefix; the other locales use a `/{locale}/{path}` prefix.

## Options Considered

1. **i18n library (Paraglide, react-intl)** — Full translation management with message catalogs, plurals, and ICU format
2. **Router rewrite pattern** — Routes don't know about locales; the router strips/adds prefixes via `rewrite` option
3. **Optional path parameter `{-$locale}`** — TanStack Router's built-in optional param syntax; routes see locale as a param

## Option Chosen

**Optional path parameter (`{-$locale}`)** with inline translation objects.

Reasons:

- No external dependencies for translations
- Type-safe locale handling via route params
- Officially documented pattern in TanStack Router
- Simple inline pattern `{ de: "...", en: "...", ru: "...", ar: "..." }[locale]` — no build step for translations
- Composable: works with any future i18n library if needed
- `LocaleString = Record<Locale, string>` lets TypeScript flag every missing translation as a compile error when a locale is added

## Implementation Details

### Locale Configuration

**File**: `src/web/utils/locale.ts`

- `Locale` type: `'de' | 'en' | 'ru' | 'ar'`
- `LOCALES` const array
- `DEFAULT_LOCALE`: `'de'`
- `RTL_LOCALES`: `['ar']`
- `isRtlLocale(locale)`: helper used by `__root.tsx` to set `<html dir>`
- `localeFromAcceptLanguage()`: Parses the `Accept-Language` header and returns the best matching locale

### `LocaleString`

**File**: `src/web/content/contentLeaf.ts`

```ts
export type LocaleString = Record<Locale, string>;
```

Every translation table in the codebase uses this shape. Widening `Locale` in `locale.ts` automatically forces every consumer to provide all
keys — that's how new locales are added safely.

### Route Structure

Uses TanStack Router's `{-$locale}` optional path parameter:

```
src/routes/
├── __root.tsx              Root shell (dynamic lang AND dir attributes)
├── {-$locale}.tsx          Layout route: validates locale, auto-redirects
├── {-$locale}/
│   └── index.tsx           Home page (matches /, /en, /ru, /ar)
└── api/                    API routes (no locale)
```

A single route file handles every locale variant. `/{-$locale}/` matches:

- `/` — locale param is undefined → defaults to `'de'`
- `/en` — locale param is `'en'`
- `/ru` — locale param is `'ru'`
- `/ar` — locale param is `'ar'`
- `/fr` — invalid → 404

### Auto-Redirect

In `src/routes/{-$locale}.tsx`, `beforeLoad` uses `createIsomorphicFn` to:

- **Server**: Read `locale` cookie, or detect from `Accept-Language` header
- **Client**: Read `locale` cookie, or fall back to default

If the detected locale differs from the default and no locale prefix is in the URL, it redirects (e.g., `/` → `/ru` for a Russian-preferring
user).

### RTL (Right-to-Left)

`src/routes/__root.tsx` sets the `dir` attribute on `<html>` based on the current locale:

```tsx
const dir = isRtlLocale(locale) ? 'rtl' : 'ltr';
return <html lang={locale} dir={dir} suppressHydrationWarning>…
```

The site's Tailwind layout uses logical-flow primitives (`flex`, `gap`, `grid`) which mirror automatically in RTL. For directional polish
use Tailwind's `rtl:` and `ltr:` variants on the few components that need them (chevrons, send-arrow icons, slide-in drawers). When adding
new components that use directional spacing, prefer logical utilities (`ms-*`/`me-*`, `start-*`/`end-*`) over
`ml-*`/`mr-*`/`left-*`/`right-*`.

Numbers, phone numbers, email addresses and Latin proper nouns embedded in Arabic prose stay LTR — the browser handles the bidi algorithm;
we do not need explicit `<bdo>` wrappers.

### Reading Locale in Components

**Hook**: `src/web/hooks/useLocale.ts`

```tsx
import { useLocale } from '../web/hooks/useLocale';

function MyComponent() {
  const locale = useLocale(); // 'de' | 'en' | 'ru' | 'ar'
  return <p>{{ de: 'Hallo', en: 'Hello', ru: 'Привет', ar: 'مرحبا' }[locale]}</p>;
}
```

### Date Formatting

**File**: `src/web/utils/dateFnsLocale.ts`

```ts
export const DATE_FNS_LOCALE: Record<Locale, DateFnsLocale> = {
  de,
  en: enUS,
  ru,
  ar,
};
```

Pass `DATE_FNS_LOCALE[locale]` to any `format()` / `formatDistanceToNow()` call so days, months and relative times render in the user's
language.

### Language Switcher

**Component**: `src/web/components/LanguageSwitcher.tsx`

Renders one entry per locale with the language's native name (Deutsch, English, Русский, العربية). On click:

1. Sets the `locale` cookie
2. Navigates to the corresponding locale path

### SEO

**File**: `src/web/seo/seoConstants.ts`

- `OG_LOCALE` maps each app locale to the IETF tag Open Graph wants (`de_DE`, `en_US`, `ru_RU`, `ar_AR`)
- `DEFAULT_SHARE_IMAGE_ALT` provides the share-image alt text per locale

`seoMeta()` emits a `<link rel="alternate" hreflang>` per configured locale plus `x-default`, so search engines map the four URL variants
together. The sitemap (`src/routes/sitemap[.]xml.ts`) iterates `LOCALES`, so it expands automatically when a locale is added.

### Chat Assistant

The visitor chat sends the active locale as part of its `assistantOptions` (see `src/web/chat/VisitorChat.graphql` and the `locale` field on
`ChatAssistantOptions`). The server agent (`agentVisitorAssistant.ts`) picks a per-locale system prompt from `LOCALE_FRAMING`, grounded in
the locale-keyed services/hours/transit text in `src/server/agents/podologieFacts.ts` (`PODOLOGIE_FACTS_LOCALIZED`). The admin chat ignores
the field and stays German.

## Adding a New Locale

1. Add the locale code to `Locale`, `LOCALES`, and (if RTL) `RTL_LOCALES` in `src/web/utils/locale.ts`
2. The TypeScript compiler will now flag every `LocaleString` / `Record<Locale, …>` literal that lacks the new key — work through the list
   with `npx tsc --noEmit` until it's clean. Hot spots are `src/web/content/*.ts`, `src/routes/{-$locale}/*.tsx` (each page's `seoMeta()`
   plus inline copy), `src/web/search/searchIndex.ts`, `src/web/components/{SiteHeader,SiteFooter,SiteSearch}.tsx`, `src/web/chat/*`, and
   `src/web/practice.ts`
3. Add the locale's entries to `OG_LOCALE` and `DEFAULT_SHARE_IMAGE_ALT` in `src/web/seo/seoConstants.ts`, and to the locale-keyed maps in
   `src/web/seo/structuredData.ts` (`LOCALE_BCP47`, `FOUNDER_JOB_TITLE`, `SERVICES_CATALOG_NAME`, `practiceDescription`, `practiceServices`)
4. Add the locale's entry to `LOCALE_NATIVE_NAME` and `TRIGGER_ARIA_LABEL` in `src/web/components/LanguageSwitcher.tsx`
5. Add the locale's date-fns entry to `DATE_FNS_LOCALE` in `src/web/utils/dateFnsLocale.ts`
6. Add the locale's entry to `LOCALE_FRAMING` and `PODOLOGIE_FACTS_LOCALIZED` so the visitor chat agent can respond in it
7. Install the corresponding cspell dictionary if available (e.g. `@cspell/dict-ru_ru`, `@cspell/dict-ar`) and reference it in
   `.cspell.json`'s `import` and `dictionaries` arrays
8. Run `npm run check` and `npm test` — all green

### Adding a New Page

Create the route file under `src/routes/{-$locale}/`:

```tsx
// src/routes/{-$locale}/about.tsx
import { createFileRoute } from '@tanstack/react-router';
import { useLocale } from '../../web/hooks/useLocale';
import { seoMeta } from '../../web/seo/seoMeta';
import { webPageUrlGet } from '../../web/seo/webPageUrlGet';
import { localeFromParam } from '../../web/utils/locale';

export const Route = createFileRoute('/{-$locale}/about')({
  head: ({ params }) => {
    const locale = localeFromParam(params);
    return seoMeta({
      title: { de: 'Über uns', en: 'About us', ru: 'О нас', ar: 'من نحن' }[locale],
      description: { de: 'Wer wir sind.', en: 'Who we are.', ru: 'Кто мы.', ar: 'من نحن.' }[locale],
      path: '/about',
      locale,
      webPageUrl: webPageUrlGet(),
    });
  },
  component: AboutPage,
});

function AboutPage() {
  const locale = useLocale();
  return <h1>{{ de: 'Über uns', en: 'About us', ru: 'О нас', ar: 'من نحن' }[locale]}</h1>;
}
```

This single file handles `/about`, `/en/about`, `/ru/about`, `/ar/about`. `seoMeta()` automatically emits `hreflang` alternates for every
configured locale plus `x-default` — see [architecture/seo.md](./seo.md). After adding the page, also add `/about` to `SITEMAP_PATHS` in
`src/web/seo/sitemapRoutes.ts` so it appears in the dynamic `/sitemap.xml`.
